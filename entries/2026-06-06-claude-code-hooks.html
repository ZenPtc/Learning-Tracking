Wiring Up Claude Code Stop Hooks

How event hooks do.

Stop Hooks

I wanted Claude Code to send me a notification the moment it finished responding.
The answer was that I needed a reaction to an event.

Claude Code has a hooks system: you register a shell command against an event name, and the harness runs it for you automatically. The event for "I just finished responding" is called Stop.

The WSL2 wrinkle

Claude Code runs inside WSL2 — a Linux VM on Windows. Linux's usual notify-send wasn't installed. Even if it were, it would show notifications inside the Linux layer, not on the Windows desktop where I actually sit.

The fix: WSL2 can call Windows binaries directly. powershell.exe is always on the path, so one PowerShell line can pop a real Windows toast notification from inside a bash script.

WSL2 trick
From WSL2 you can run any .exe on your Windows PATH — including powershell.exe. This bridges the OS gap: a bash script that talks directly to Windows.

How it was built

1
Test the notification command first

Ran the PowerShell balloon-notification command directly in the terminal to confirm it actually popped a toast before wiring anything up (exit code 0 = success).

2
Save it as a small script

Put the PowerShell invocation in ~/.claude/claude-done-notify.sh instead of cramming it into JSON.

A .sh file is just a plain text file full of shell commands — the .sh extension is a convention that tells you "this is a shell script." By default Linux won't run it as a program, so chmod +x was needed: it grants the file execute permission, meaning the OS is now allowed to treat it as something you can run directly.

3
Register the Stop hook in global settings

Edited ~/.claude/settings.json — the user-level file so the hook applies to every project, not just one. The relevant part that was added looks like this:

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/claude-done-notify.sh",
            "async": true
          }
        ]
      }
    ]
  }
}
Why async: true?
The notification runs in the background. Claude's response is never held up waiting for the script to finish.
4
Validate the JSON

Used Python (python3 -c "import json, sys; json.load(sys.stdin)") to confirm the file is valid — since jq wasn't installed.

"Valid" here means the JSON is correctly formatted: every opening bracket has a closing bracket, every comma is in the right place, all quotes are closed. If even one character is wrong, the computer can't parse the file at all and Claude Code would silently ignore the entire settings file.

The event flow

🤖 Claude responds
Turn ends
↓
⚡ Stop event
Harness fires hook
↓
📜 Bash script
Calls powershell.exe
↓
🔔 Toast notification
Windows tray

The one thing you can't verify mid-turn

The Stop hook fires after the turn ends — outside Claude's own execution. So the only live proof is the notification that appeared (or didn't) when the previous message finished. The tool is real; the test is live.

If it didn't fire
Run /hooks in Claude Code to reload hook config without restarting, or close and reopen the app. New settings.json entries sometimes need a fresh session.
